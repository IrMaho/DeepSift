# 🌐 Universal Workspace — Agent Workflow Rules

## 🧭 پروتکل شروع هر مکالمه (قانون طلایی)

### مرحله ۱: کشف خودکار بافت پروژه (Auto-Discovery)

در اولین مکالمه با هر پروژه جدید، قبل از هر اقدامی:

1. **اسکن ریشه پروژه:** ساختار دایرکتوری را بخوان و معماری پروژه (MVC, Clean Architecture, Monorepo, Microservice, Component-based, یا هر الگوی دیگر) را شناسایی کن.
2. **شناسایی زبان و فریم‌ورک:** از روی فایل‌های تنظیمات ریشه (مانند `package.json`, `pubspec.yaml`, `Cargo.toml`, `go.mod`, `pom.xml`, `*.csproj`, `Makefile`, `pyproject.toml` یا هر فایل مشابه) زبان، فریم‌ورک و ابزار بیلد پروژه را تشخیص بده.
3. **جستجوی سند مرجع (اگر وجود داشت):** به دنبال اسنادی مانند فایل‌های معماری، نقشه راه، Handoff یا Blueprint بگرد. **اگر پیدا شد**، آن سند مرجع نهایی پروژه محسوب می‌شود و انحراف از آن ممنوع است. **اگر پیدا نشد**، بدون اشاره به نبود آن، مستقیم سراغ درخواست کاربر برو.
4. **جستجوی فایل پیشرفت (اگر وجود داشت):** اگر فایلی مثل `progress.md`, `todo.md`, `CHANGELOG.md` یا مشابه آن وجود داشت، بخوان و اولین تسک انجام‌نشده `[ ]` را اعلام کن. **اگر نبود**، بدون اشاره به نبود آن ادامه بده.
5. **پروتکل هشدار مغایرت:** اگر سند مرجعی پیدا شد و درخواست کاربر با آن در تضاد بود، قبل از اقدام هشدار بده و تاییدیه صریح بگیر.

## 📁 الگوهای ساختاری

* قبل از ساخت فایل جدید، الگوی نام‌گذاری و ساختار پوشه‌بندی موجود پروژه را شناسایی و رعایت کن.
* اگر پروژه از الگوی خاصی پیروی می‌کند (مثلاً feature-first, layer-first, domain-driven)، همان الگو را ادامه بده.

## ⚙️ قوانین ثابت کدنویسی (Zero-Violation Rules)

| قانون | توضیح |
| --- | --- |
| ❌ صفر کامنت | هیچ‌گونه کامنت توضیحی در کد مجاز نیست. کد باید خود-توضیح‌دهنده باشد. |
| ✅ سقف خطوط | فایل‌های ماژولار/کامپوننتی حداکثر ۲۰۰ خط. فایل‌های سرویسی/منطقی حداکثر ۵۰۰ خط. |
| ✅ ماژولار | هر فایل مسئول یک وظیفه مشخص. تفکیک صریح مسئولیت‌ها (SoC). |
| ✅ تایپ‌سیف | قبل از تعریف تایپ جدید، تایپ‌های موجود پروژه را بررسی کن. تعاریف داده‌ای باید صریح و دقیق باشند. |
| ✅ DRY | قبل از ساخت هر تابع، کلاس یا المان جدید، مطمئن شو نمونه مشابهی در پروژه وجود ندارد. |

## 🎨 قوانین طراحی (فقط در صورت وجود لایه UI)

اگر پروژه دارای رابط کاربری است، این قوانین را رعایت کن:

* **توکن‌ها:** اگر پروژه سیستم توکن استایل دارد (فایل متغیرها، theme، design tokens)، تمام مقادیر بصری باید از آن بیایند. مقادیر hardcoded ممنوع.
* **چندزبانگی:** اگر پروژه سیستم i18n دارد، هیچ متن نمایشی hardcoded مجاز نیست.
* **تم‌ها:** اگر پروژه از تم‌های چندگانه پشتیبانی می‌کند، عناصر UI باید در تمام تم‌ها درست نمایش داده شوند.
* **استفاده مجدد:** قبل از ساخت المان بصری جدید، کامپوننت‌های مشترک موجود در پروژه را بررسی کن.

---

## 📋 فرایند پیاده‌سازی هر تسک

1. **تطابق با نقشه راه:** اگر سند مرجعی وجود دارد، تسک باید با آن هماهنگ باشد.
2. **بررسی مدل‌های داده:** تایپ‌ها، اینترفیس‌ها و مدل‌های موجود پروژه را قبل از تعریف جدید بررسی کن.
3. **بررسی ابزارهای موجود:** توابع کمکی، سرویس‌ها و utilityهای موجود را قبل از بازنویسی بررسی کن.
4. **پیاده‌سازی تمیز:** رعایت سقف خطوط، بدون کامنت، ماژولار.
5. **بیلد/کامپایل:** پس از هر تغییر، دستور بیلد پروژه (هر چه که باشد) را اجرا کن تا خطایی نباشد.
6. **ردیابی پیشرفت:** اگر فایل پیشرفت وجود دارد، تسک تکمیل‌شده را تیک بزن.

## 🧪 چک‌لیست پایان هر مکالمه

* آیا پروژه بدون خطا بیلد/کامپایل می‌شود؟
* آیا فایل‌ها از سقف خطوط مجاز تجاوز نمی‌کنند؟
* آیا کامنتی در کد وجود ندارد؟
* آیا کد تکراری ساخته نشده؟
* آیا مقادیر hardcoded (در استایل یا متون نمایشی) وجود ندارد؟ (فقط اگر پروژه سیستم توکن/i18n دارد)
